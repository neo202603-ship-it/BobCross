# 밥크로스 앱 완성도 전문가 평가 보고서

작성일: 2026-06-13

## 평가 범위

- 앱: 밥크로스(Bab-Cross)
- 패키지: `com.babcross.app`
- 현재 코드 기준 버전: `0.1.167 / versionCode 168`
- 검토 자료: `README.md`, `CHANGELOG.md`, `app/build.gradle.kts`, `AndroidManifest.xml`, `MainActivity.kt`, Nearby/protocol/store 계층, Play Store 제출 문서, 기존 개선/보안/카드 아이디어 문서

## 한 줄 결론

밥크로스는 "가까운 사람들과 서버 없이 메뉴를 정한다"는 핵심 사용 장면을 실제 내부 테스트 가능한 수준까지 끌어올린 베타 후반 제품이다. 기능, 브랜드, 신뢰 장치는 이미 초기 앱 평균을 넘지만, 대중 출시 관점에서는 Nearby 연결 실패를 덜 불안하게 만드는 UX, 첫 30초의 즉시 이해, 반복 사용 동기, 앱 밖 공유/초대 전환율이 다음 완성도 병목이다.

## 전문가별 평가

### 1. 제품 전략 전문가

밥크로스의 가장 큰 장점은 "무엇을 해결하는 앱인지"가 분명하다는 점이다. 점심, 회식, 음료, 후식처럼 작지만 반복되는 선택 피로를 잡고 있고, 이를 단순 투표가 아니라 "밥판을 열고 밥신호를 보내는 놀이형 결정"으로 포장했다.

강점:

- 반복 빈도가 높은 문제를 다룬다.
- 서버 없이 가까운 사람끼리 쓴다는 정체성이 명확하다.
- 혼자 체험하고 결과 카드까지 만들 수 있어 "상대가 없어도 앱이 빈손이 되는 문제"가 줄었다.
- 최근 버전에서 늦게 들어오는 참여자와 연결 수 표시를 보강해 실제 사용 흐름에 더 가까워졌다.

위험:

- 사용자는 이미 카카오톡 투표, 사다리, 룰렛 웹앱이라는 쉬운 대체재를 갖고 있다.
- "모두가 앱을 설치해야 한다"는 마찰이 매우 크다.
- Nearby라는 기술 차별점이 사용자에게는 가치가 아니라 권한 요청과 연결 불확실성으로 느껴질 수 있다.

판단:

초기 포지셔닝은 "메뉴 추천 앱"보다 "같은 장소에 모인 사람들의 빠른 밥결정판"이 더 강하다. 대중 앱으로 바로 넓히기보다 직장 동료, 스터디, 동아리, 가족 외식처럼 반복적으로 같이 먹는 소규모 그룹을 첫 시장으로 잡는 편이 현실적이다.

### 2. UX 리서처

사용자 여정은 크게 `첫 실행 -> 밥판 열기 -> 참여/선택 -> 결과 확인 -> 공유/기록`으로 이어진다. 현재 앱은 각 단계의 기능을 갖췄고, 첫 실행 온보딩, 권한 없이 체험하기, 프로필 칩, 밥판 열기 작성 카드, 결과 공유 미리보기까지 사용자의 망설임을 줄이는 장치가 다수 들어가 있다.

강점:

- 홈에서 `밥판 열기`와 `밥판 체험하기`가 바로 보인다.
- 권한이 부담스러운 사용자를 체험 흐름으로 돌릴 수 있다.
- 밥판 작성 화면이 종류, 후보, 템플릿, 룰렛, 규칙으로 정돈되어 있다.
- 결과 화면에서 지도 검색, 공유, 템플릿 저장, 다시 열기 등 다음 행동이 있다.
- 설정에 개인정보처리방침과 로컬 데이터 관리가 있어 신뢰 회복 동선이 있다.

위험:

- 첫 사용자는 "주변 연결", "Nearby", "권한", "밥판 코드"를 한 번에 받아들여야 한다.
- 연결이 0명일 때 성공 상태인지 실패 상태인지 혼란스러울 수 있다.
- 밥판장과 참여자의 화면 차이를 사용자가 사전에 예측하기 어렵다.
- 기능이 많아진 만큼 처음 보는 사람에게는 재미와 복잡함이 동시에 온다.

권장:

1. 첫 밥판 열기 직전 `상대에게 이렇게 보여요` 미니 프리뷰를 더 강하게 노출한다.
2. 연결 0명 상태를 실패가 아니라 `혼자 먼저 정리 가능`과 `밥친구 기다리는 중`으로 분명히 나눈다.
3. 초대받은 사람 화면에는 `내 선택 공개 여부`, `후보 추가 가능 여부`, `남은 시간`을 한 카드로 고정한다.

### 3. UI/브랜드 디자이너

밥크로스는 브랜드 언어가 앱 전반에 살아 있다. `밥판`, `밥신호`, `밥친구`, `밥닉`, `밥해시`, `오늘의 밥결정` 같은 용어는 기능을 설명하면서도 기억성을 만든다. 결과 카드는 특히 강한 차별점이다.

강점:

- 앱 이름과 마스코트, 결과 카드가 일반 투표 앱과 확실히 구분된다.
- 프로필 칩과 아바타 노출은 "내가 어떻게 보이는지"를 즉시 알려준다.
- 결과 카드 미리보기와 공유 이미지는 앱 밖에서 브랜드를 전달하는 핵심 자산이다.
- Play Store 그래픽, 아이콘, 스크린샷 산출물이 이미 준비되어 있다.

위험:

- 밥크로스 자체 캐릭터보다 Bob Ross 패러디 감성이 앞서 보이면 공개 마켓에서 법적/브랜드 리스크가 남는다.
- 설정/진단 영역에서 기술 용어가 남으면 브랜드 언어의 몰입이 깨질 수 있다.
- 결과 카드가 강한 만큼 밥판 작성/초대 화면의 시각적 재미가 상대적으로 기능적으로 보일 수 있다.

권장:

1. 장기적으로는 패러디 의존을 줄이고 밥크로스 마스코트 자체의 이름, 표정, 카드 테마를 축적한다.
2. 오류/권한/진단 문구도 `밥친구를 못 찾을 때`, `밥신호 다시 찾기`처럼 브랜드 언어로 통일한다.
3. 결과 카드 공유 이미지에는 안전 영역과 긴 텍스트 축약 규칙을 계속 엄격하게 유지한다.

### 4. Android 엔지니어

기술적으로는 단일 Activity 기반의 로컬 앱이지만, 구현 범위가 매우 넓다. Nearby Connections, 로컬 저장소, 메시지 envelope, replay guard, 결과 해시, 공유 이미지 생성, 온보딩, 커스텀 UI, 히스토리, 템플릿, 룰렛, 지도 인텐트까지 한 앱 안에 들어가 있다.

강점:

- `minSdk 26`, `targetSdk 35`, `compileSdk 35`로 최신 Play 기준에 맞춰져 있다.
- release 빌드에 R8 minify/resource shrink가 명시되어 있다.
- FileProvider가 비공개 provider로 설정되어 있고 공유 경로가 제한되어 있다.
- Nearby 메시지에 schema version, messageId, createdAtMillis가 들어가며 replay guard가 존재한다.
- store 계층에서 손상 JSON을 복구하는 `runCatching` 방어가 들어가 있다.

위험:

- `MainActivity.kt`가 화면, 상태, 프로토콜 처리, 검증, 이미지 생성까지 지나치게 많이 담당한다.
- 다기기 Nearby 플로우는 코드가 맞아도 실제 OS/권한/제조사/전원 정책에 따라 깨질 수 있다.
- 연결 역할 전환, 지각 참여자, 중복 초대, 종료 타이밍은 회귀가 생기기 쉬운 영역이다.
- UI가 코드로 작성되어 있어 화면별 회귀 테스트와 스냅샷 확인이 어렵다.

권장:

1. `MainActivity.kt`에서 Nearby 메시지 처리와 데이터 검증을 `protocol` 또는 `domain` 계층으로 분리한다.
2. 연결 상태 계산, 투표 종료 가능 여부, 결과 생성 로직은 순수 Kotlin 단위 테스트 가능한 형태로 빼낸다.
3. 회귀가 잦은 늦은 참여자/연결 수/종료 알림 시나리오를 자동 테스트 또는 수동 체크리스트로 고정한다.

### 5. 보안/개인정보 전문가

현재 보안 방향은 꽤 좋다. 자체 서버, 광고 SDK, 분석 SDK가 없고, Android Auto Backup 비활성화, cleartext traffic 차단, FileProvider 제한, 로컬 데이터 삭제, 개인정보처리방침, Nearby 메시지 검증이 들어가 있다.

강점:

- `android:allowBackup="false"`와 `android:usesCleartextTraffic="false"`가 적용되어 있다.
- 앱 내부 개인정보처리방침이 서버 미수집과 주변 기기 전달 범위를 분리해 설명한다.
- 로컬 데이터 관리에서 지난 결정, 영수증/해시, 템플릿, 프로필/기본값, 전체 삭제를 나눠 제공한다.
- payload 크기, senderId 불일치, 손상 JSON, 해시 불일치 등에 대한 방어가 문서화/구현되어 있다.
- Android 13+ `NEARBY_WIFI_DEVICES`에 `neverForLocation` 플래그가 적용되어 있다.

위험:

- Nearby는 근거리 참여자를 신뢰하는 모델이므로 악성 참여자나 조작 메시지에 대한 완전 방어가 어렵다.
- 사용자는 "서버 저장 없음"을 "아무에게도 전달되지 않음"으로 오해할 수 있다.
- Play Console Data safety 표현과 앱 내부 문구, 개인정보처리방침이 버전 증가 후에도 계속 동기화되어야 한다.

권장:

1. 초대/결과 화면에 `주변 참여자 기기에 전달됨`, `서버 저장 없음`, `상대 기기에 전달된 데이터는 회수 불가`를 짧게 유지한다.
2. release AAB 기준 `debuggable=false`, R8 산출물, Nearby 실제 플로우를 매 릴리스 체크한다.
3. 악성 payload, 오래된 메시지, 중복 messageId, hash mismatch 테스트를 계속 늘린다.

### 6. QA 리드

밥크로스의 품질은 단일 화면보다 상태 조합에서 결정된다. 권한 허용/거부, Bluetooth/Wi-Fi/위치 상태, 연결 전/후, 밥판장/참여자, 공개/비공개 선택, 제한시간 종료/수동 종료, 늦은 참여자 등 조합 폭이 크다.

강점:

- 내부 테스트 문서와 단말 체크리스트가 이미 있다.
- S24 Ultra, S21, Fold 계열 등 여러 삼성 기기 테스트 기록이 있다.
- 스크린샷 산출물과 Play 내부 테스트 등록 기록이 남아 있다.
- 최근 변경은 실제 현장 시나리오인 늦은 참여자, 연결 수 동기화, 종료 알림을 집중 개선했다.

위험:

- 최신 코드 버전 `0.1.167`와 Play 내부 테스트 업로드 버전 `168 (0.1.167)`가 일치한다.
- 현장 회귀는 보통 "성공 경로"가 아니라 권한 거부, 연결 끊김, 중간 재진입, 화면 회전/앱 재시작에서 난다.
- fold/tablet 화면은 스크린샷은 있지만 실제 상호작용 회귀가 계속 필요하다.

권장 테스트 매트릭스:

1. 1명 단독: 권한 없음, 체험, 혼자 결과 카드 생성
2. 2명 기본: 생성, 참여, 투표, 수동 종료, 공유
3. 3명 이상: 지각 참여자, 거절자, 미응답자, 모두 선택 완료 알림
4. 권한 실패: 위치 거부, Bluetooth off, Wi-Fi off, Nearby Wi-Fi 거부
5. 데이터 상태: 앱 데이터 삭제 후 첫 실행, 손상/빈 기록, 템플릿 없음
6. 결과 상태: 동률, 무득표, 비공개 선택, 긴 후보명, 후보 10개 이상

### 7. Play Store 출시 컨설턴트

출시 준비는 꽤 많이 진행되어 있다. 개인정보처리방침 URL, Play 설명 초안, Data safety 방향, 권한 설명, 스크린샷, feature graphic, 내부 테스트 트랙 기록이 있다. 최신 앱 버전과 내부 테스트 트랙도 `168 (0.1.167)` 기준으로 맞췄으므로, 다음 병목은 테스터 관찰과 폐쇄 테스트 전환 준비다.

강점:

- Play 등록 문구가 상황 중심으로 작성되어 있다.
- Data safety에서 개발자 서버 미수집과 Nearby 전달을 분리하려는 방향이 좋다.
- 앱 아이콘, feature graphic, phone/fold/tablet 스크린샷 산출물이 있다.
- 내부 테스트 배포 경험이 이미 있다.

위험:

- Android-only 앱이라 실제 그룹 사용 가능성이 제한된다.
- 권한 설명이 조금만 모호해도 사용자와 심사자 모두에게 위치 추적 앱처럼 보일 수 있다.
- 스토어 첫 화면에서 "왜 설치해야 하는지"가 기술보다 상황으로 보여야 한다.

권장:

1. 최신 버전 기준으로 Play 문서의 release candidate와 릴리스 노트를 갱신한다.
2. 스크린샷 1장은 반드시 `메뉴 못 정하는 팀 -> 밥판 열기 -> 결과 카드`의 전환을 보여준다.
3. 짧은 설명은 `가까운 사람들과 메뉴 후보를 올리고 빠르게 고르는 근거리 밥판 앱`처럼 현재 방향을 유지한다.

### 8. 성장/마케팅 전문가

밥크로스는 광고형 대중 확산보다 그룹 내 전파가 더 자연스럽다. 한 명이 앱을 켜고 결과 카드를 대화방에 던졌을 때, 나머지가 "다음엔 나도 참여해볼까"라고 느끼는 구조를 만들어야 한다.

강점:

- 결과 카드가 공유 소재가 된다.
- 밥크로스 용어는 대화방에서 농담처럼 퍼질 수 있다.
- 점심/회식/카페/야식/후식 등 사용 상황을 확장하기 쉽다.
- 카드 대결 아이디어는 재방문성과 밥결정 이후 놀이를 키울 여지가 있다.

위험:

- 참여자가 앱 설치 전 결과 카드만 보면 "재미있는 이미지"로 끝날 수 있다.
- 공유 카드에 앱 설치/참여 이유가 약하면 바이럴 루프가 닫힌다.
- 카드 대결을 너무 빨리 크게 만들면 핵심 메뉴 결정 흐름이 무거워진다.

권장:

1. 공유 카드 하단에 `다음 밥판은 같이 고르기` 정도의 짧은 초대 문구를 넣는다.
2. 앱 미설치자도 공유 이미지만 보고 후보/득표/이유를 이해하게 한다.
3. 카드 대결은 우선 `동률 결승전` 또는 `오늘의 밥카드 점수`처럼 결과 후 보너스로 작게 시작한다.

### 9. 데이터/리텐션 분석가

서버와 분석 SDK가 없는 구조이므로 개인정보 측면은 좋지만, 제품 개선에 필요한 사용 데이터가 부족하다. 따라서 초기에는 원격 분석보다 로컬/수동 지표와 테스트 관찰을 잘 설계하는 편이 맞다.

봐야 할 지표:

- 첫 실행 후 `밥판 체험하기` 완료율
- 첫 실행 후 실제 `밥판 열기` 도달률
- 밥판 생성 후 참여자 1명 이상 연결 비율
- 밥판 생성 후 결과 카드 생성 비율
- 결과 카드 공유 시도율
- 지난 결정에서 `다시 열기` 사용률
- 권한 거부 후 체험으로 전환하는 비율

권장:

1. 외부 서버 분석을 넣기 전, 내부 테스트에서는 수동 관찰표로 위 지표를 기록한다.
2. 앱 안에는 사용자에게 보이는 개인정보 친화형 기록, 예를 들어 `이번 주 밥결정 횟수`와 `자주 나온 메뉴`를 강화한다.
3. 리텐션 기능은 알림보다 `지난 결정으로 다시 열기`, `2등 메뉴 재도전`, `밥카드`처럼 앱 안의 자연스러운 재방문 이유를 먼저 만든다.

### 10. 법무/IP 리스크 검토자

밥크로스는 이름, 문구, 시각 연출에서 패러디적 재미가 있다. 내부 테스트와 개인 프로젝트에서는 강한 매력 요소지만, 공개 마켓과 장기 브랜드 관점에서는 특정 인물/방송/작품을 직접 떠올리게 하는 표현을 줄이는 편이 안전하다.

위험:

- README의 "전설의 화가 이름을 살짝 비튼" 설명은 공개 자료에서는 불필요한 리스크 신호가 될 수 있다.
- "메뉴 고르기 참 쉽죠?" 같은 문구가 특정 밈/인물을 강하게 연상시킬 수 있다.
- 카드 대결 확장 시 특정 카드게임의 명칭/규칙/레이아웃을 직접 차용하면 별도 리스크가 생긴다.

권장:

1. 공개 스토어 문구에서는 패러디 출처를 설명하지 않는다.
2. 앱 내부 문구는 `밥크로스 자체 세계관` 중심으로 서서히 교체한다.
3. 카드 대결은 `밥카드`, `밥력`, `단결력`, `결정권` 등 자체 명칭을 사용한다.

## 현재 완성도에서 가장 좋은 점

1. 핵심 루프가 실제 앱처럼 이어진다: 생성, 초대, 참여, 투표, 결과, 공유, 기록이 모두 있다.
2. 브랜드 언어가 살아 있다: 기능명 자체가 제품의 재미를 만든다.
3. 신뢰 기반이 좋다: 서버 없음, 로컬 데이터 관리, 개인정보 설명, 메시지 방어가 있다.
4. 최근 개선이 실사용 문제에 집중되어 있다: 연결 수 동기화, 늦은 참여자, 모두 선택 완료 알림, 작성 화면 정리가 특히 중요하다.
5. 출시 산출물이 있다: Play 문서, 스크린샷, 아이콘, 내부 테스트 기록이 이미 준비됐다.

## 현재 완성도에서 가장 아쉬운 점

1. 첫 사용자가 권한과 Nearby 연결을 이해해야 하는 부담이 남아 있다.
2. 설치 전 사용자를 설득하는 공유/초대 루프가 아직 약하다.
3. `MainActivity.kt`에 너무 많은 책임이 몰려 있어 유지보수 리스크가 커지고 있다.
4. 최신 코드 버전과 Play 내부 테스트 버전은 맞췄으나, 폐쇄 테스트 전환 전 실제 테스터 관찰이 더 필요하다.
5. Android-only와 다기기 연결 변수가 시장 확산의 가장 큰 제약이다.

### P0. 연결 실패/0명 상태 UX 완성

Nearby 앱의 품질은 연결 실패를 얼마나 덜 무섭게 보여주는지에 달려 있다. 현재 앱은 체험과 혼자 결과 생성이 가능하므로, 0명 상태를 실패가 아니라 두 갈래 행동으로 보여줘야 한다.

제안 문구:

- `지금은 나만 참여 가능해요`
- `밥친구를 기다리거나, 내 선택으로 먼저 결과 카드를 만들 수 있어요`
- `권한 없이 밥판 체험하기`
- `밥친구 다시 찾기`

### P0. 초대받은 사람의 규칙 카드 고정

참여자는 앱 구조보다 "내가 무엇을 누르면 되는지"와 "내 선택이 보이는지"가 중요하다.

필수 정보:

- 보낸 밥친구
- 후보 수
- 남은 시간
- 내 선택 공개 여부
- 후보 추가 가능 여부
- 참여 후 변경 가능 여부

### P1. 공유 카드의 초대 기능 강화

결과 카드는 밥크로스의 바이럴 자산이다. 공유 카드가 예쁜 결과물에서 끝나지 않고 다음 참여를 부르는 초대장이 되어야 한다.

제안:

- 카드 하단에 `다음 밥판은 같이 골라요` 문구 추가
- 공유 문구에 후보/득표/이유/앱명 포함
- 결과 카드 미리보기에서 `이미지 공유`, `문구만 복사`, `같은 후보로 다시 열기` 유지

### P1. 핵심 로직 분리와 테스트 보강

기능이 충분히 많아졌으므로 이제는 코드 구조가 제품 속도를 좌우한다.

분리 후보:

- Nearby 메시지 검증
- 투표 종료 가능 여부 계산
- 연결 표시 계산
- 결과 생성 및 동률 처리
- 공유 카드 텍스트 생성

테스트 우선순위:

- 늦은 참여자 연결 수 반영
- 모두 선택 완료 알림 반복 조건
- 거절자/미응답자/무득표 결과
- 공개/비공개 선택 결과 검증
- 중복/오래된 Nearby 메시지 무시

### P2. 반복 사용 이유 강화

초기 설치 후 한 번 써보고 끝나지 않게 하려면 기록 화면이 행동 화면이 되어야 한다.

제안:
