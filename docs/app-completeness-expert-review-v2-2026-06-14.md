밥크로스는 1차 평가 시점보다 "현장 사용 중 헷갈리는 연결 상태"와 "핵심 판단 로직의 테스트 가능성"이 개선되어 베타 후반 완성도가 한 단계 올라갔다. 다만 로컬 최신 버전 `0.1.184`와 Play 제출 문서의 릴리스 후보 `0.1.167` 사이에 간극이 생겼고, `MainActivity.kt`가 1만 줄을 넘은 구조 리스크는 다음 출시 이후 반드시 줄여야 한다.

## 전문가 패널 평가

### 1. 제품 전략 전문가

2차 기준 밥크로스는 "근거리 밥판"이라는 포지션이 더 또렷해졌다. 특히 자동 연결이 꺼진 상태에서도 QR 참여와 밥신호 보내기 같은 명시적 행동은 연결 의도로 보고 흐름을 이어가게 한 점이 제품적으로 맞다. 사용자는 설정 원칙보다 "방금 내가 하려던 일"이 막히지 않는 것을 더 중요하게 느낀다.

강점:

- 메뉴 결정이라는 반복 문제를 매우 구체적인 상황으로 잡고 있다.
- 혼자 체험, 실제 밥판, QR/코드 참여, 결과 카드까지 제품 루프가 완성되어 있다.
- 0명 상태를 실패가 아니라 진행 가능한 상태로 다루려는 방향이 좋다.
- 브랜드 언어가 기능명을 대체해 설명 비용을 줄인다.

위험:

- 대체재는 여전히 강하다. 카카오톡 투표, 룰렛 웹앱, 말로 정하기보다 명확히 편해야 한다.
- 앱 설치 전 사용자에게는 Nearby의 장점보다 권한 요청이 먼저 보인다.
- Play 문서가 로컬 최신 버전을 따라가지 못하면 내부 테스트/공개 테스트 운영 신뢰가 흔들릴 수 있다.

판단:

출시 직전에는 새 기능보다 "첫 밥판 성공률"이 가장 중요하다. 제품 지표도 다운로드 수보다 첫 실행 후 실제 밥판 열기, 2인 이상 참여, 결과 카드 공유까지 이어지는 비율을 중심으로 봐야 한다.

### 2. UX 리서처

현재 UX의 가장 큰 개선은 연결 상태를 숫자 경쟁처럼 보이지 않게 하고, 사용자가 조치할 수 있는 상태에서만 버튼을 주려는 흐름이다. 연결 램프가 상태 중심으로 정리된 것은 Nearby 앱에서 매우 중요한 변화다.

강점:

- 홈, 밥판 열기, 체험, 설정, 결과 공유로 이어지는 주요 동선이 갖춰져 있다.
- 자동 연결이 꺼진 상태에서 명시적 참여/발행 의도는 흐름을 계속 이어가도록 조정됐다.
- 연결 램프 팝업이 정상/대기 상태에서 불필요한 액션을 줄이는 방향으로 정리됐다.
- 홈의 아바타/밥닉 표시와 워드마크가 시스템 글자 크기 영향에 덜 흔들리게 개선됐다.

위험:

- 첫 사용자가 권한 요청 전에 "왜 필요한지"를 완전히 이해하지 못하면 이탈한다.
- QR, 4자리 코드, Nearby 자동 연결, 밥판장/참여자 역할이 한 번에 들어오면 복잡하게 느껴질 수 있다.
- 기능이 많아진 만큼 홈에서 가장 중요한 첫 행동이 계속 압도적으로 보여야 한다.

권장:

1. 권한 요청 직전에는 기술명이 아니라 `가까운 밥친구를 찾기 위해 필요해요` 수준의 상황 문구를 유지한다.
2. 참여자 화면 상단에는 남은 시간, 공개 여부, 후보 추가 가능 여부를 항상 같은 위치에 둔다.
3. 연결 램프 팝업은 지금처럼 "상태 설명 + 필요한 경우만 조치" 원칙을 계속 지킨다.

### 3. UI/브랜드 디자이너

밥크로스는 일반 메뉴 추천 앱보다 브랜드 자산이 강하다. 특히 밥판, 밥신호, 밥친구, 밥닉, 결과 카드의 조합은 앱 밖에서 말로 전파되기 쉽다. 2차 기준 워드마크와 프로필 칩 정리는 작은 변경처럼 보여도 첫 화면의 완성도를 올리는 효과가 크다.

강점:

- 기능명과 브랜드명이 같은 세계관 안에서 움직인다.
- 결과 카드와 공유 문구가 바이럴 소재로 쓸 수 있는 형태다.
- 아바타/밥닉네임 표시가 자기 표현과 참여감을 만든다.
- 연결 램프는 숫자보다 상태를 빠르게 읽게 만드는 좋은 방향이다.

위험:

- 장식적 재미가 많아질수록 실제 결정 완료까지의 속도를 해치면 안 된다.
- 결과 카드가 강한 만큼 작성/초대 화면도 같은 수준의 polish를 유지해야 한다.
- 외부 공개 문구에서 특정 패러디 출처가 떠오르는 표현은 계속 줄이는 편이 안전하다.

권장:

1. Play 스크린샷 첫 장은 앱 기능 설명보다 "메뉴 못 정하는 상황이 밥판으로 정리되는 장면"을 보여준다.
2. 결과 카드 공유 이미지의 하단 초대 문구는 짧고 반복적으로 유지한다.
3. 오류/권한/진단 문구도 브랜드 언어로 바꾸되, 사용자가 해야 할 행동은 모호하게 만들지 않는다.

### 4. Android 아키텍트

기술 구조는 1차 대비 분명히 좋아졌다. `PollCompletionPolicy`, `PollParticipationPolicy`, `SharedResultInsights`, `NearVoteMessageValidator`가 생겨 핵심 판단 일부가 테스트 가능한 순수 Kotlin 로직으로 빠졌다. 이 변화는 회귀 위험이 큰 Nearby 투표 앱에서 매우 의미 있다.

강점:

- `compileSdk 35`, `targetSdk 35`, `minSdk 26`으로 현재 Play 기준 대응이 좋다.
- release 빌드에 minify/resource shrink가 적용되어 있다.
- Nearby 메시지에 schema, senderId, messageId, createdAtMillis, payload length 검증 흐름이 있다.
- 참여 현황과 모두 선택 완료 판단이 테스트 가능한 정책 객체로 분리됐다.
- 결과 공유 문구/동률/무득표 판단도 `SharedResultInsights`로 분리되어 있다.

위험:

- `MainActivity.kt`가 10,647줄로 UI, 상태, Nearby 처리, 공유 이미지, 세션 복원, 권한, 애니메이션을 여전히 과도하게 담당한다.
- 순수 정책 분리는 시작됐지만 메시지 처리 orchestration은 여전히 Activity 내부 집중도가 높다.
- UI가 코드 기반이라 화면 회귀를 사람이 계속 봐야 한다.

권장:

1. 다음 구조 개선 1순위는 `handleNearbyMessage`와 메시지별 처리 함수를 Activity 밖으로 옮기는 것이다.
2. 권한/기기 설정 상태 계산을 `ConnectionReadinessPolicy` 같은 순수 계층으로 분리한다.
3. 결과 카드 렌더링 텍스트 계산과 실제 Canvas/View 렌더링을 더 분리해 긴 텍스트 회귀를 줄인다.

### 5. 보안/개인정보 전문가

보안/개인정보 방향은 강점이다. 서버 없음, 광고/분석 SDK 없음, `allowBackup=false`, `usesCleartextTraffic=false`, 제한된 FileProvider, Nearby payload 검증은 초기 앱치고 매우 좋은 기반이다.

강점:

- 앱이 개발자 서버로 데이터를 보내지 않는 구조가 명확하다.
- Nearby로 전달되는 데이터와 로컬 저장 데이터의 범위를 문서화하고 있다.
- 메시지 크기, senderId, schema, timestamp, replay 조건을 검증한다.
- 손상 JSON을 복구하거나 무시하는 방어 흐름이 있다.
- Android 13+ `NEARBY_WIFI_DEVICES`에 `neverForLocation`을 사용한다.

위험:

- "서버 저장 없음"이 "상대 기기에 전달되지 않음"으로 오해될 수 있다.
- 근거리 참여자 자체가 악의적이면 조작 메시지와 혼란을 완전히 막기 어렵다.
- Play Data safety, 개인정보처리방침, 앱 내부 문구는 버전이 올라갈 때마다 맞춰야 한다.

권장:

1. 결과/초대 화면의 신뢰 문구는 `서버 저장 없음`, `주변 참여자 기기에 전달`, `상대 기기 데이터는 회수 불가` 세 문장을 계속 짧게 유지한다.
2. 오래된 메시지, 미래 메시지, 중복 messageId, oversized payload, sender mismatch 테스트를 릴리스 체크에 고정한다.
3. 공개 테스트 전 release AAB의 `debuggable=false`와 R8 산출물을 다시 확인한다.

### 6. QA 리드

품질의 핵심은 "기능 존재 여부"가 아니라 다기기 상태 조합이다. 2차 기준 단위 테스트가 늘어난 점은 좋지만, Nearby 제품은 결국 실제 기기 조합 테스트가 품질을 결정한다.

강점:

- 정책 로직 테스트가 늘어 회귀를 더 빨리 잡을 수 있다.
- 연결 수/참여 현황/모두 선택 완료 알림 같은 실제 현장 이슈를 계속 다루고 있다.
- 단독 체험, QR 참여, 코드 참여, 수동 종료, 제한시간 종료까지 시나리오 폭이 넓다.

위험:

- 로컬 최신 버전 `0.1.184`는 1차 당시 내부 테스트 검증 버전 `0.1.167`보다 많이 앞서 있다.
- 연결 램프와 자동 연결 정책은 UX는 좋아졌지만 상태 조합 테스트가 다시 필요하다.
- 권한 거부, Bluetooth/Wi-Fi off, 위치 off, 앱 재시작, 화면 회전, 데이터 삭제 후 복원은 계속 회귀 위험이 높다.

권장 테스트 매트릭스:

1. 단독: 권한 없이 체험, 혼자 밥판 열기, 결과 카드 공유
2. 2기기: 코드 참여, QR 참여, 투표, 수동 종료, 결과 수신
3. 3기기 이상: 지각 참여자, 미응답자, 거절자, 모두 선택 완료 알림
4. 권한/설정 실패: Bluetooth off, Wi-Fi off, 위치 off, Nearby Wi-Fi 거부, 앱 설정 이동
5. 데이터 상태: 앱 데이터 삭제 후 첫 실행, 손상 세션, 템플릿 삭제, 지난 결정 삭제
6. 텍스트 극단값: 긴 질문, 긴 후보명, 후보 10개 이상, 큰 글씨/굵은 글씨

### 7. Play Store 출시 컨설턴트

현재 가장 큰 출시 운영 리스크는 앱 코드보다 버전 정렬이다. 로컬 CHANGELOG와 Gradle은 `0.1.184 / versionCode 185`까지 진행됐지만, `docs/play-store-submission.md`는 릴리스 후보와 Play 내부 테스트 상태가 `0.1.167 / versionCode 168` 기준이다.

강점:

- Play 설명, Data safety 방향, 권한 설명, 개인정보처리방침 URL, 스크린샷 산출물이 준비되어 있다.
- 스토어 문구가 기술보다 상황 중심으로 작성되어 있다.
- 내부 테스트 배포 경험이 있고, 기존 AAB/스크린샷/그래픽 산출물이 정리되어 있다.

위험:

- 최신 로컬 버전과 Play 문서가 불일치하면 어느 버전이 심사 후보인지 팀 내 판단이 흐려진다.
- 연결 램프/자동 연결 정책처럼 최근 변경은 실제 단말 검증 후 릴리스 노트에 반영되어야 한다.
- 권한 설명이 조금만 모호하면 위치 추적 앱처럼 보일 수 있다.

권장:

1. 공개/폐쇄 테스트로 올릴 후보를 `0.1.184`로 할지, 별도 안정화 버전으로 올릴지 먼저 정한다.
2. 후보가 정해지면 `docs/play-store-submission.md`의 release candidate, 릴리스 노트, 단말 테스트 결과를 같은 버전으로 갱신한다.
3. Play Console 실제 상태는 업로드 전후 라이브로 다시 확인한다.

### 8. 성장/마케팅 전문가

밥크로스의 확산은 광고보다 그룹 내 반복 사용에서 나온다. 한 사람이 밥판을 열고 결과 카드를 공유했을 때, 다음 식사 때 다른 사람이 "이번엔 나도 들어가 볼까"라고 느껴야 한다.

강점:

- 결과 카드와 `다음 밥판은 같이 고르기` 문구가 공유 루프의 씨앗이다.
- 점심, 회식, 음료, 후식, 간식 등 반복 상황으로 확장하기 쉽다.
- 밥친구/밥닉/밥신호 같은 말맛이 대화방에서 퍼지기 쉽다.

위험:

- 설치 전 사용자는 공유 이미지만 보고 앱의 실사용 가치를 놓칠 수 있다.
- Android-only라 iPhone 사용자가 섞인 그룹에서는 곧바로 전체 참여가 어렵다.
- 설치, 권한, Nearby 연결이라는 세 단계 마찰이 전환율을 낮출 수 있다.

권장:

1. 공유 문구는 후보/결과/앱명/다음 행동을 한 번에 담되 너무 길게 만들지 않는다.
2. "결과 카드만 받아본 사람"을 위한 스토어 첫 스크린샷을 강화한다.
3. 초기 타깃은 회사 점심팟, 동아리, 스터디, 가족 외식처럼 Android 사용자가 반복적으로 모이는 그룹이 좋다.

### 9. 데이터/리텐션 분석가

분석 SDK가 없는 것은 개인정보 신뢰 면에서 장점이지만, 제품 판단에는 별도 관찰 설계가 필요하다. 초기에는 원격 분석보다 내부 테스트 관찰표가 더 적합하다.

관찰해야 할 지표:

- 첫 실행 후 체험 시작률
- 체험 완료 후 실제 밥판 열기 진입률
- 밥판 생성 후 참여자 1명 이상 연결 비율
- QR/코드 참여 성공률
- 투표 시작 후 결과 생성 완료율
- 결과 카드 공유 시도율
- 지난 결정에서 다시 열기 사용률
- 권한 거부 후 재시도 또는 체험 전환율

권장:

1. 내부 테스트 참가자에게 "막힌 순간"을 시간순으로 적게 한다.
2. 지표는 정량보다 실패 이유 태깅부터 시작한다.
3. 앱 내 리텐션은 알림보다 `지난 2등 메뉴 재도전`, `같은 후보로 다시 열기`, `이번 주 밥결정`을 먼저 키운다.

### 10. 법무/IP 리스크 검토자

1차와 동일하게, 장기 공개 운영에서는 특정 인물/작품/카드게임을 직접 떠올리게 하는 표현을 줄이는 편이 안전하다. 최근 README가 밥크로스 자체 설명으로 정리된 점은 긍정적이다.

강점:

- 현재 README는 `Bab` 신호와 `Cross`를 중심으로 자체 의미를 설명한다.
- 앱 기능명 대부분이 자체 세계관으로 작동한다.
- 카드 대결 아이디어를 핵심 루프 밖 보너스로 유지하려는 방향이 안전하다.

위험:

- 외부 홍보 문구에서 과거 패러디 설명을 다시 쓰면 리스크가 되살아난다.
- 카드형 UI가 특정 카드게임의 명칭, 레이아웃, 규칙을 직접 연상시키면 별도 리스크가 생긴다.

권장:

1. 공개 스토어/홍보 문구에서는 패러디 출처를 설명하지 않는다.
2. 카드 확장은 `밥카드`, `밥력`, `단결력`, `오늘의 밥결정`처럼 자체 명칭을 쓴다.
3. 스크린샷과 feature graphic에서도 제3자 브랜드/밈을 떠올리는 표현은 피한다.

### 11. 접근성/포용성 전문가

2차에서 워드마크와 연결 램프가 시스템 글자 크기 영향을 덜 받도록 조정된 점은 시각 안정성에 도움이 된다. 다만 직접 그리는 UI가 늘수록 접근성 텍스트, 터치 영역, 색상 외 구분이 빠지기 쉽다.

강점:

- 큰 글씨/굵은 글씨 영향을 줄이려는 렌더링 보정이 있다.
- 연결 램프 색상 체계가 단순해져 빠른 인지가 가능하다.
- 권한/설정 안내를 조치 가능한 상태 중심으로 보여주려는 방향이 좋다.

위험:

- 색상만으로 연결 상태를 구분하면 색각 이상 사용자에게 불리할 수 있다.
- 직접 그리는 워드마크/램프/카드 UI는 TalkBack 설명이 부족할 수 있다.
- 긴 후보명과 큰 글씨 조합은 결과 카드 공유 이미지에서 계속 깨질 가능성이 있다.

권장:

1. 연결 램프에는 색상 외 상태 텍스트 또는 contentDescription을 반드시 유지한다.
2. 주요 액션 버튼은 최소 터치 영역과 TalkBack 라벨을 확인한다.
3. 스크린샷 QA에 큰 글씨, 굵은 글씨, 다크 모드, 화면 확대를 포함한다.

### 12. 운영/지원 전문가

초기 테스트에서 사용자 문의는 대부분 기능 설명보다 "왜 연결이 안 되나요?"에 집중될 가능성이 높다. 지원 비용을 줄이려면 앱 내부에서 실패 원인과 해결 행동을 바로 보여줘야 한다.

강점:

- 설정에 권한/개인정보/로컬 데이터 관리 동선이 있다.
- 연결 램프 팝업이 상태 진단의 중심이 될 수 있다.
- README와 Play 문서에 두 기기 테스트 절차가 정리되어 있다.

위험:

- Nearby 연결 실패는 앱 버그, OS 권한, 제조사 전원 정책, 거리 문제, Wi-Fi/Bluetooth 상태가 섞여 원인 파악이 어렵다.
- 내부 테스트 버전과 로컬 최신 버전이 다르면 사용자 제보를 재현하기 어려워진다.

권장:

1. 테스트 배포마다 "이 버전에서 확인할 것"을 3개 이하로 정한다.
2. 연결 안 됨 문의용 체크리스트를 앱 내부 또는 docs에 짧게 유지한다.
3. 버그 제보에는 앱 버전, 기기 모델, Android 버전, 권한 상태, 연결 램프 상태를 함께 받는다.

## 2차 우선순위 개선안

### P0. 릴리스 후보 확정 순서 정리

릴리스 후보 버전은 가장 먼저 정할 일이 아니라, 남은 수정 범위를 닫고 회귀 테스트를 통과한 뒤 마지막에 확정해야 한다. 초반에는 `0.1.184`를 임시 기준점으로만 두고, 이후 수정이 발생하면 후보 버전은 자연스럽게 올라간다. 최종 후보가 정해진 뒤 Play 제출 문서, 릴리스 노트, 테스트 체크리스트, 실제 AAB 산출물의 버전을 한 번에 맞춘다.

완료 기준:

- 남은 수정 범위와 동결 기준이 먼저 정리됨
- 회귀 테스트 중 수정이 발생하면 후보 버전 확정을 뒤로 미룸
- `docs/play-store-submission.md`의 release candidate가 최신 후보와 일치
- CHANGELOG 상단 버전과 Gradle `versionName/versionCode` 일치
- release AAB 파일, 해시, 단말 설치 버전 기록 갱신
- Play Console 업로드 전후 실제 상태 확인

### P0. 최신 연결 UX 회귀 테스트

0.1.171부터 0.1.184까지 연결 램프, 자동 연결, QR 참여, 밥신호 흐름이 집중적으로 바뀌었다. UX 방향은 맞지만 회귀 테스트가 반드시 필요하다.

완료 기준:

- 자동 연결 켜짐/꺼짐 각각에서 밥신호 보내기 테스트
- QR 참여가 자동 연결 꺼짐 상태에서도 의도대로 이어지는지 확인
- 연결 램프의 초록/노랑/빨강/회색 상태별 팝업 확인
- 조치가 필요 없는 상태에서 불필요한 버튼이 없는지 확인

### P1. Activity 책임 추가 분리

이미 정책 분리는 좋은 방향으로 시작됐다. 다음은 Activity 내부의 Nearby 메시지 처리와 연결 준비 상태 계산을 분리해야 한다.

분리 후보:

- `NearbyMessageHandler`: poll/response/profile/receipt/result/connection sync 처리
- `ConnectionReadinessPolicy`: 권한, Bluetooth, Wi-Fi, 자동 연결 상태 문구와 액션 계산
- `ResultCardTextPolicy`: 화면 카드/공유 이미지/공유 문구의 긴 텍스트 규칙
- `InviteCodePolicy`: 코드/QR 만료, host token, 입력 상태 계산

### P1. 접근성 QA 추가

직접 그리는 UI가 많아진 만큼 TalkBack과 큰 글씨 QA를 별도 체크로 둔다.

완료 기준:

- 홈 주요 버튼, 연결 램프, 밥판 열기, QR 참여, 결과 공유 버튼 TalkBack 라벨 확인
- 큰 글씨/굵은 글씨/화면 확대에서 버튼 텍스트 겹침 확인
- 연결 상태가 색상만이 아니라 텍스트/라벨로도 구분됨

### P2. 공유 루프 고도화

현재 `다음 밥판은 같이 고르기` 문구는 좋은 출발점이다. 다음 단계는 공유 이미지를 본 사람이 앱을 설치해야 하는 이유를 즉시 이해하게 만드는 것이다.

완료 기준:

- 공유 문구에 앱명, 결과, 다음 행동이 들어감
- Play 첫 스크린샷이 공유 카드와 같은 메시지를 이어받음
- 결과 카드에서 `같은 후보로 다시 열기`가 반복 사용 행동으로 잘 보임

## 최종 판단

밥크로스의 2차 완성도는 "초기 아이디어를 넘어 실제 테스트 가능한 제품"에서 "제한적 공개 테스트를 준비할 수 있는 제품"으로 올라섰다. 특히 최근 개선은 눈에 띄는 새 기능보다 연결 상태, 자동 연결, 정책 분리처럼 사용 중 불안을 줄이는 방향이라 좋다.

다만 다음 단계는 단호해야 한다. 출시 전에는 기능을 더 늘리기보다 남은 수정 범위를 닫고, 실제 기기 회귀와 접근성 확인을 끝낸 뒤, 마지막에 최종 후보 버전과 Play 문서를 동기화해야 한다. 출시 후에는 `MainActivity.kt`의 책임을 계속 줄여야 밥크로스가 빠르게 좋아지면서도 덜 깨지는 앱이 된다.
