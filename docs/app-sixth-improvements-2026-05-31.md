# 밥크로스 6차 개선/보완안

작성일: 2026-05-31

## 현재 기준

- 최신 로컬 배포 버전: `0.1.60 / versionCode 61`
- 직전 개선 문서: `docs/app-fifth-improvements-2026-05-31.md`
- 출시 메모: `docs/play-store-submission.md`
- 앱 구조: 단일 Android 앱, 패키지 `com.babcross.app`, Nearby Connections 기반 근거리 밥판

5차 이후 밥크로스는 단순 점심 메뉴 투표를 넘어, `종류 선택 → 종류별 템플릿/룰렛 → 후보 정리 → 결과 카드 → 지도/공유/다시 열기`까지 이어지는 제품 루프를 갖췄다. 6차는 새 기능을 크게 늘리기보다, 출시 전 반복 사용성과 데이터 일관성, 보안/개인정보 설명, Google Play 등록 준비도를 끌어올리는 보완 단계로 둔다.

## 서브 역할 검토 방식

이번 개선안은 다음 역할로 나눠 검토한 뒤 취합했다.

| 역할 | 검토 초점 | 핵심 결론 |
| --- | --- | --- |
| UX/UI | 첫 진입, 작성 화면 밀도, 초대/선택/결과 흐름 | 작성 화면이 기능적으로 풍부해졌으나 첫 진입과 기본/고급 설정 분리가 필요하다. |
| 음식데이터/템플릿 | 카테고리, 룰렛, 칼로리, 기본/사용자 템플릿 | 음식 데이터가 MainActivity, 칼로리 카탈로그, Store에 흩어져 있어 단일 소스화가 필요하다. |
| 보안/개인정보 | 권한, Nearby payload 검증, 로컬 저장, 신뢰 모델 | 기존 보호 장치는 늘었지만 권한 매트릭스, 메시지 버전, 로컬 데이터 관리가 남았다. |

## 6차 핵심 목표

```text
더 빨리 열고, 더 믿고 쓰고, 더 안전하게 출시할 수 있는 밥판으로 정리한다.
```

판단 기준은 다음과 같다.

- 홈에서 밥판 작성까지 가는 탭 수가 줄었는가
- 작성 화면에서 처음 사용자도 무엇만 하면 되는지 알 수 있는가
- 음식 카테고리, 룰렛, 칼로리, 템플릿 데이터가 서로 어긋나지 않는가
- Nearby로 오가는 데이터의 검증과 버전 호환성이 명확한가
- Google Play Data safety, 권한 설명, 개인정보처리방침이 같은 말을 하는가
- 내부 테스트/비공개 베타에 올릴 수 있는 릴리스 후보를 고정할 수 있는가

## 1단계: 밥판 작성 첫 진입 단축

현재 홈의 `밥판 열기`는 먼저 종류 선택 화면을 거친다. 종류를 고정한 것은 메뉴 혼합을 막는 데 효과적이지만, 반복 사용자에게는 한 단계가 길게 느껴질 수 있다.

제안:

- 최근 사용한 종류 또는 기본 종류를 저장한다.
- 홈의 `밥판 열기`는 저장된 종류로 바로 작성 화면에 진입한다.
- 작성 화면 상단 종류 카드에는 `다른 종류로 시작` 보조 액션을 둔다.
- 최초 설치 또는 저장된 종류가 없을 때만 종류 선택 화면을 먼저 보여준다.

기대 효과:

- 반복 사용자는 홈에서 곧바로 질문/후보 입력을 시작한다.
- 종류 선택은 유지하되 매번 강제하지 않는다.
- “음료수 밥판”처럼 최근 맥락을 앱이 기억하는 느낌이 생긴다.

검증 포인트:

- 신규 설치 첫 사용자는 종류 선택을 반드시 거치는가
- 두 번째 밥판부터 최근 종류로 바로 들어가는가
- 템플릿/룰렛은 여전히 현재 종류로 고정되는가
- 최근 종류 삭제/초기화 시 기본값이 안전한가

## 2단계: 작성 화면 기본/고급 분리

현재 작성 화면에는 질문, 후보, 룰렛, 후보 추가 허용, 선택 공개, 제한시간, 템플릿 저장, 상대 프리뷰가 모두 노출된다. 기능은 좋지만 처음 쓰는 사람에게는 “무엇만 하면 되는지”가 흐려질 수 있다.

제안:

- 기본 영역: 종류, 종류별 템플릿, 오늘의 질문, 메뉴 후보, 룰렛, 생성 버튼
- 고급 영역: 제한시간, 후보 추가 가능, 선택 공개 여부, 템플릿 저장
- `상대에게 이렇게 보여요` 프리뷰는 고급 영역 아래가 아니라 규칙 요약과 붙인다.
- 고급 영역은 기본 접힘 상태 또는 `밥판 규칙` 카드로 묶는다.

기대 효과:

- 빠른 작성 사용자는 질문/후보만 보고 밥판을 만들 수 있다.
- 설정을 바꾸고 싶은 사용자는 명확한 영역에서 조정한다.
- 템플릿 저장이 최종 확인 흐름을 방해하지 않는다.

검증 포인트:

- 기본값만으로 밥판 생성까지 흐름이 막히지 않는가
- 고급 설정 변경 후 프리뷰와 실제 Nearby payload가 일치하는가
- 긴 질문/후보 10개 이상/작은 화면에서 버튼이 겹치지 않는가

## 3단계: 음식 데이터 단일 소스화

현재 음식 데이터는 세 곳에 나뉘어 있다.

- 카테고리/룰렛 후보: `MainActivity`
- 칼로리: `MenuCalorieCatalog`
- 기본 템플릿: `NearVoteStore`

이 구조는 메뉴를 하나 추가할 때 카테고리, 칼로리, 템플릿, 추론 로직을 모두 수동으로 맞춰야 한다. 이미 음료/후식/간식 룰렛과 칼로리 보강 과정에서 누락 위험이 드러났다.

제안:

- `MenuCatalog`를 새로 두고 메뉴명, 카테고리, 별칭, kcal, 룰렛 노출 여부, 템플릿 사용 여부를 한곳에서 관리한다.
- 룰렛 후보, 칼로리 추정, 카테고리 추론은 모두 `MenuCatalog`를 참조한다.
- `패스`, `아무거나`, `미정` 같은 비음식 후보는 명시적 예외로 관리한다.
- 칼로리는 `약 N kcal`에 더해 “1인분 기준 추정” 문구 또는 낮음/보통/높음 구간을 제공한다.

기대 효과:

- 새 메뉴 추가 시 누락이 줄어든다.
- 결과 → 템플릿 저장 → 재사용 시 카테고리 손실이 줄어든다.
- 칼로리 힌트가 건강 정보처럼 과하게 정확해 보이는 위험을 줄인다.

검증 포인트:

- 모든 룰렛 후보가 카테고리와 칼로리 힌트를 갖는가
- 모든 기본 템플릿 후보가 카탈로그 또는 예외 목록에 있는가
- 별칭 메뉴도 올바른 카테고리와 칼로리로 추론되는가
- `PollTemplate` JSON round-trip에서 `categoryKey`가 유지되는가

## 4단계: 기본 템플릿과 룰렛 균형 보강

현재 기본 템플릿은 식사/요리에 치우쳐 있고, 음료는 적으며 후식/간식 기본 템플릿은 비어 있을 수 있다. 카테고리 선택에는 다섯 종류가 모두 노출되므로, 모든 종류에서 첫 화면이 비지 않아야 한다.

제안:

- 음료 템플릿 최소 3개
  - `카페 주문 정하기`
  - `시원한 음료 고르기`
  - `회의 전 한 잔`
- 후식 템플릿 최소 3개
  - `밥 먹고 달달하게`
  - `카페 디저트 고르기`
  - `더운 날 후식`
- 간식 템플릿 최소 3개
  - `편의점 간식`
  - `야근 전 간식`
  - `분식 간식`
- 음료/후식/간식 룰렛 후보를 각각 12~20개 수준으로 확장한다.
- 같은 카테고리 안 고정 룰렛 묶음끼리 후보가 과도하게 중복되지 않게 한다.

기대 효과:

- 사용자가 어떤 종류를 골라도 바로 쓸 수 있는 템플릿이 보인다.
- `아무거나 3개` 반복 체감이 줄어든다.
- 밥크로스가 “점심 메뉴 앱”을 넘어 “선택 도구”로 확장된다.

검증 포인트:

- 신규 설치 상태에서 모든 카테고리 템플릿 화면이 비지 않는가
- 룰렛 반복 10회 이상에서 같은 세트가 과하게 반복되지 않는가
- 템플릿 선택 후 질문/후보/제한시간/공개 설정이 자연스럽게 채워지는가

## 5단계: Nearby 보안/호환성 강화

현재 앱은 payload 크기 제한, JSON 파싱 방어, senderId 불일치 방어, 결과 해시 검증, 로컬 저장 제한을 갖췄다. 6차에서는 출시 전 방어선을 더 명확히 만든다.

제안:

- `NearVoteMessage`에 `schemaVersion`, `messageId`, `createdAtMillis`를 추가한다.
- wrapper 레벨에서 senderId 길이, payloadJson 길이, 허용 type을 먼저 검사한다.
- PROFILE, POLL_RESPONSE, VOTE에도 이름 길이, ID 길이, avatarId 범위 검증을 적용한다.
- `revealSelections=false` 결과에 상세 선택이 포함되어 오면 폐기한다.
- 권한 매트릭스를 Android 8~15 기준으로 정리한다.

기대 효과:

- 구버전/신버전 메시지 혼재에 대비할 수 있다.
- 악의적이거나 깨진 payload를 더 앞단에서 거를 수 있다.
- Play 심사와 사용자 문의에 답할 근거가 명확해진다.

검증 포인트:

- 64KB 초과 payload 무시
- malformed JSON 무시
- 알 수 없는 type 무시
- senderId 불일치 무시
- 후보 수/문자 길이 초과 무시
- 음수 득표/중복 participantId/hash 불일치 결과 무시
- 비공개 결과에 선택 상세가 있으면 저장하지 않음

## 6단계: 개인정보와 로컬 데이터 관리

앱은 서버, 광고 SDK, 분석 SDK 없이 동작한다. 하지만 밥닉, 아바타, 후보, 선택 결과는 로컬 저장되거나 Nearby로 주변 참여자에게 전달된다. “수집하지 않는다”와 “주변 기기에 전달된다”를 계속 분리해서 설명해야 한다.

제안:

- 설정에 `내 로컬 데이터 관리`를 추가한다.
- 삭제 단위:
  - 지난 결정
  - 영수증/해시
  - 사용자 템플릿
  - 프로필/기본값
  - 전체 초기화
- 개인정보처리방침 내부 시행일과 외부 문서 시행일을 맞춘다.
- “내 기기에서 삭제해도 이미 상대 기기에 전달된 밥판 데이터는 회수되지 않는다”를 명시한다.

기대 효과:

- 사용자에게 데이터 통제감이 생긴다.
- Google Play Data safety와 개인정보처리방침의 일관성이 좋아진다.
- 근거리 공유 모델을 숨기지 않는 신뢰 톤이 유지된다.

검증 포인트:

- 각 삭제 액션이 실제 SharedPreferences 상태를 지우는가
- 삭제 후 홈/설정/결과 화면이 빈 상태로 안전하게 돌아가는가
- 개인정보처리방침, 권한 전 안내, Play 등록 메모가 같은 표현을 쓰는가

## 7단계: Google Play 내부 테스트 준비

현재 Gradle은 `targetSdk=35`로, Google Play의 2026년 기준 신규 앱/업데이트 제출 요구인 Android 15(API 35) 이상 조건에 부합한다. Google 공식 문서는 신규 앱과 업데이트가 Android 15(API 35) 이상을 대상으로 해야 한다고 안내한다. 또한 Play Console 릴리스 전에는 스토어 등록정보, 앱 콘텐츠, 가격/배포 설정을 준비해야 하며, Data safety 양식은 모든 Play 배포 앱에 필요하다.

공식 참고:

- Google Play target API level requirement: https://developer.android.com/google/play/requirements/target-sdk
- Prepare and roll out a release: https://support.google.com/googleplay/android-developer/answer/9859348
- Data safety form: https://support.google.com/googleplay/android-developer/answer/10787469
- Prepare your app for release: https://developer.android.com/studio/publish/preparing
- Google Play policies timeline: https://developer.android.com/distribute/play-policies

제안:

- `docs/play-store-submission.md`의 버전을 현재 릴리스 후보 기준으로 갱신한다.
- 최신 코드 기준 `:app:bundleRelease`를 재생성한다.
- AAB의 `versionName`, `versionCode`, 서명, `debuggable=false`를 확인한다.
- 개인정보처리방침 URL은 GitHub blob보다 공개 접근이 안정적인 정적 페이지를 검토한다.
- Play 등록용 짧은 설명, 긴 설명, 권한 설명, 릴리스 노트를 한 문서에 모은다.
- 스크린샷 5장을 확정한다.

스크린샷 시나리오:

1. 홈: 메뉴 못 정할 때 밥판 열기
2. 밥판 열기: 종류별 템플릿과 룰렛으로 후보 만들기
3. 초대/참여: 이 밥판 규칙 보고 바로 고르기
4. 결과 카드: 오늘의 식사/요리/음료수 결정
5. 신뢰 안내: 서버 저장 없음, 주변 기기 공유 범위

검증 체크리스트:

- `testDebugUnitTest`, `compileDebugKotlin`, `bundleRelease` 통과
- 두 기기 이상에서 생성, 초대 수락, 투표, 조기 종료, 시간 만료, 결과 공유 확인
- 권한 거부, Bluetooth 꺼짐, Wi-Fi 꺼짐, 위치 꺼짐 상태 확인
- 앱 내부 개인정보처리방침과 Play Data safety 답변 불일치 확인
- 릴리스 후보 커밋 고정 후 AAB 산출물 보관

## 8단계: QA 매트릭스

6차부터는 “작동한다”보다 “어떤 조건에서 확인했다”를 남기는 것이 중요하다.

| 영역 | 조건 |
| --- | --- |
| 화면 크기 | 소형 폰, 일반 폰, 폴더블 내부/외부 화면 |
| 글꼴 | 기본, 큰 글꼴, 최대 글꼴 |
| 후보 | 2개, 5개, 10개 이상, 긴 메뉴명 |
| 질문 | 짧은 질문, 긴 질문, 이모지 포함 질문 |
| 권한 | 전체 허용, Nearby 거부, 위치 꺼짐, Bluetooth 꺼짐, Wi-Fi 꺼짐 |
| 연결 | 1:1, 1:N, 중복 연결, 연결 실패 후 재시도 |
| 결과 | 단독 1등, 동률, 무득표, 비공개 선택, 후보 추가 허용 |
| 공유 | 이미지 공유, 문구 복사, 지도 앱 없음, 외부 앱 없음 |
| 데이터 | 앱 재시작, 앱 데이터 삭제, 손상된 템플릿 JSON, 결과 20개 초과 |

## 6차 제안 실행 순서

1. 밥판 작성 첫 진입 단축
2. 작성 화면 기본/고급 분리
3. 음식 데이터 단일 소스화
4. 기본 템플릿/룰렛 균형 보강
5. Nearby 메시지 버전/검증 강화
6. 로컬 데이터 관리와 개인정보 문구 정렬
7. Google Play 내부 테스트 산출물 정리
8. QA 매트릭스 기반 단말 검증

## 이번 문서에서 의도적으로 미룬 것

- 서버 기반 동기화
- 위치 기반 맛집 추천
- 사용자 계정/로그인
- 광고/분석 SDK 도입
- iOS 또는 웹 버전
- 영양 정보의 공식 데이터베이스 연동
